## Introduction to Cosmos SDK Stores

Cosmos SDK applications persist state through a multistore. The multistore is a store of stores: each module gets its own substore, and the application composes them into one root state tree.

At a high level:

* `CommitMultiStore` is the root store mounted by the application.
* `CacheMultiStore` is an in-memory branch used while executing state transitions.
* `KVStore` is the standard byte-oriented key-value store modules interact with.
* `ObjKVStore` stores Go objects directly, mainly used for efficiency in BlockSTM enabled applications.
* Wrappers such as `cachekv`, `gaskv`, `prefix`, and `listenkv` extend store behavior without changing module code.

```text expandable
+-----------------------------------------------------+
|                                                     |
|    +--------------------------------------------+   |
|    |                                            |   |
|    |  KVStore 1 - Managed by Module 1 keeper    |   |
|    |                                            |   |
|    +--------------------------------------------+   |
|                                                     |
|    +--------------------------------------------+   |
|    |                                            |   |
|    |  KVStore 2 - Managed by Module 2 keeper    |   |
|    |                                            |   |
|    +--------------------------------------------+   |
|                                                     |
|    +--------------------------------------------+   |
|    |                                            |   |
|    |  KVStore 3 - Managed by Module 3 keeper    |   |
|    |                                            |   |
|    +--------------------------------------------+   |
|                                                     |
|                    Main Multistore                  |
|                                                     |
+-----------------------------------------------------+

                   Application State
```

Each substore is accessed through a store key, usually declared in the module's `module.go` file.

For most developers, the important distinction is between the store package's internal layers and the APIs you actually use in module code:

* App and runtime code mount stores, load versions, configure pruning, and manage the root `CommitMultiStore`.
* Module keepers work with a `KVStoreService`, `MemoryStoreService`, or `TransientStoreService`.
* Lower-level SDK code can also access stores directly from `sdk.Context` with `KVStore`, `TransientStore`, and `ObjectStore`.

### How Module Code Gets a Store

Most modules do not receive a raw `CommitMultiStore`. Instead, they receive a store service that can open the module's store for a given context:

```go expandable
type Keeper struct {
	storeService store.KVStoreService
	cdc          codec.BinaryCodec
}

func NewKeeper(storeService store.KVStoreService, cdc codec.BinaryCodec) Keeper {
	return Keeper{
		storeService: storeService,
		cdc:          cdc,
	}
}

func (k Keeper) Example(ctx context.Context) {
	store := k.storeService.OpenKVStore(ctx)
}
```

If you are working directly with `sdk.Context`, the context already exposes gas-metered store accessors:

```go expandable
func (c Context) KVStore(key storetypes.StoreKey) storetypes.KVStore {
	return gaskv.NewStore(c.ms.GetKVStore(key), c.gasMeter, c.kvGasConfig)
}

func (c Context) TransientStore(key storetypes.StoreKey) storetypes.KVStore {
	return gaskv.NewStore(c.ms.GetKVStore(key), c.gasMeter, c.transientKVGasConfig)
}
```

This is why module code usually sees a `KVStore` or `ObjKVStore` instead of raw commit stores and multistores.

### Store Interface

A store exposes its type and supports cache branching:

```go expandable
package types

type Store interface {
	GetStoreType() StoreType
	CacheWrapper
}

type Committer interface {
	Commit() CommitID
	LastCommitID() CommitID
	WorkingHash() []byte

	SetPruning(pruningtypes.PruningOptions)
	GetPruning() pruningtypes.PruningOptions
}

type CommitStore interface {
	Committer
	Store
}
```

The important split is between:

* `Store`, which can be branched with `CacheWrap()`.
* `Committer`, which can persist state.
* `CommitStore`, which combines both.

If you are writing normal keeper logic, you rarely need to think in terms of `Store` and `CommitStore`. These interfaces become relevant when you are implementing a new store backend or a wrapper such as `cachekv`, `gaskv`, or `prefix`.

### Commit Store

A commit store is any store that can flush its current working state and return a deterministic `CommitID`. In practice, commit capability is deliberately restricted. Module code typically works with `KVStore`, not `CommitKVStore`, so only the application root can finalize state changes.

`CommitID` is what ultimately contributes to the application hash included in block headers. `WorkingHash()` exposes the hash of the current working state before commit.

## Multistore

### Multistore Interface

Every Cosmos SDK application keeps a multistore at its root.

```go expandable
package types

type MultiStore interface {
	Store

	CacheMultiStore() CacheMultiStore
	CacheMultiStoreWithVersion(version int64) (CacheMultiStore, error)

	GetStore(StoreKey) Store
	GetKVStore(StoreKey) KVStore
	GetObjKVStore(StoreKey) ObjKVStore

	LatestVersion() int64
}

type CacheMultiStore interface {
	MultiStore
	Write()
}
```

Application code mounts stores into the multistore with store keys. Module code generally does not interact with `MultiStore` directly after wiring is complete. Instead, the multistore is the layer that routes a store key to the correct `KVStore`, `ObjKVStore`, or memory-backed store.

`GetKVStore` is the byte-oriented path most keepers use. `GetObjKVStore` exists for object stores, which allow callers to store Go values directly instead of manually marshaling bytes. This should only be used as a Transient store.

### CommitMultiStore

`CommitMultiStore` is the root multistore that persists state and manages store loading, upgrades, snapshots, pruning, and rollback.

```go expandable
package types

type CommitMultiStore interface {
	Committer
	MultiStore
	snapshottypes.Snapshotter

	EarliestVersion() int64

	MountStoreWithDB(key StoreKey, typ StoreType, db dbm.DB)

	LoadLatestVersion() error
	LoadLatestVersionAndUpgrade(upgrades *StoreUpgrades) error
	LoadVersionAndUpgrade(ver int64, upgrades *StoreUpgrades) error
	LoadVersion(ver int64) error

	SetInterBlockCache(MultiStorePersistentCache)
	SetInitialVersion(version int64) error
	SetIAVLCacheSize(size int)
	SetIAVLDisableFastNode(disable bool)
	SetIAVLSyncPruning(sync bool)
	RollbackToVersion(version int64) error

	ListeningEnabled(key StoreKey) bool
	AddListeners(keys []StoreKey)
	PopStateCache() []*StoreKVPair
}
```

As for concrete implementation, the default root multistore is `rootmulti.Store`:

```go expandable
package store

func NewCommitMultiStore(db dbm.DB, logger log.Logger) types.CommitMultiStore {
	return rootmulti.NewStore(db, logger)
}
```

In practice, `CommitMultiStore` is primarily for app authors and framework code. Its main responsibilities are:

* mounting substores with the right `StoreKey` and `StoreType`
* loading historical or latest versions
* committing the new version at the end of block execution
* configuring pruning, inter-block caching, and state listeners
* coordinating snapshots and rollback

Module code should stay one layer above this and depend on store services or context store accessors.

### CacheMultiStore

`cachemulti.Store` is the in-memory branch type used during execution. It wraps each mounted substore in a cache layer and defers writes until `Write()` is called:

```go expandable
package cachemulti

type Store struct {
	stores map[types.StoreKey]types.CacheWrap
}

func (cms Store) Write() {
	for _, store := range cms.stores {
		store.Write()
	}
}
```

This is the mechanism that makes execution safe and efficient: code mutates a branch first, and writes only flow to the parent stores if execution succeeds. `CacheMultiStore` is what gives the SDK a transactional working view of state during block and transaction processing.

Most keepers will not instantiate a `CacheMultiStore` themselves, but it is important to understand that the store you interact with during execution is usually a cached branch, not the final committed state.

## Base-layer KVStores

### `KVStore` and `CommitKVStore` Interfaces

The store package is generic internally and then aliases the byte-oriented forms used by most modules:

```go expandable
package types

type GBasicKVStore[V any] interface {
	Get(key []byte) V
	Has(key []byte) bool
	Set(key []byte, value V)
	Delete(key []byte)
}

type GKVStore[V any] interface {
	Store
	GBasicKVStore[V]

	Iterator(start, end []byte) GIterator[V]
	ReverseIterator(start, end []byte) GIterator[V]
}

type (
	Iterator     = GIterator[[]byte]
	BasicKVStore = GBasicKVStore[[]byte]
	KVStore      = GKVStore[[]byte]

	ObjIterator     = GIterator[any]
	ObjBasicKVStore = GBasicKVStore[any]
	ObjKVStore      = GKVStore[any]
)

type CommitKVStore interface {
	Committer
	KVStore
}
```

From a module author's perspective, `KVStore` is the core abstraction:

* use `Get`, `Has`, `Set`, and `Delete` for point operations
* use `Iterator` and `ReverseIterator` for ordered scans
* use `prefix.Store` when you want to work inside a subspace of your module store

One important iterator rule still applies: no writes should happen within an iterator's domain while the iterator exists, except for the cache store implementation that is explicitly designed to handle that pattern safely.

### `IAVL` Store

The default persistent byte-oriented store mounted by `rootmulti.Store` is the IAVL-backed store:

```go expandable
package iavl

func LoadStore(
	db dbm.DB,
	logger log.Logger,
	key types.StoreKey,
	id types.CommitID,
	cacheSize int,
	disableFastNode bool,
) (types.CommitKVStore, error) {
	return LoadStoreWithInitialVersion(db, logger, key, id, 0, cacheSize, disableFastNode)
}

func (st *Store) Commit() types.CommitID {
	hash, version, err := st.tree.SaveVersion()
	if err != nil {
		panic(err)
	}

	return types.CommitID{
		Version: version,
		Hash:    hash,
	}
}
```

IAVL stores are based on the [IAVL Tree](https://github.com/cosmos/iavl), which provides:

* `O(log n)` reads and writes.
* Ordered iteration over key ranges.
* Versioned immutable snapshots of prior state, subject to pruning settings.

Application developers do not instantiate `iavl.Store` directly. Instead, `rootmulti.Store` mounts it for persistent key-value substores, and modules interact with it through the higher-level store interfaces.

### `DbAdapter` Store

`dbadapter.Store` adapts a raw `dbm.DB` to the `KVStore` interface:

```go expandable
package dbadapter

type Store struct {
	dbm.DB
}

func (Store) GetStoreType() types.StoreType {
	return types.StoreTypeDB
}

func (dsa Store) CacheWrap() types.CacheWrap {
	return cachekv.NewStore(dsa)
}
```

This adapter is mostly used where a plain database needs to participate in store interfaces without implementing the whole stack itself.

### `Transient` Store

A transient store is reset after each commit. It is useful for per-block state that should not survive across heights.

```go expandable
package transient

type GStore[V any] struct {
	internal.BTreeStore[V]
}

type Store struct {
	GStore[[]byte]
}

type ObjStore struct {
	GStore[any]
}

func (ts *GStore[V]) Commit() (id types.CommitID) {
	ts.Clear()
	return id
}
```

Because `Commit()` clears the in-memory tree, transient state is automatically discarded at the end of the block. This makes transient stores a good fit for temporary indexes, block-scoped bookkeeping, and state that is needed during execution but should never become part of consensus state.

The store package also supports memory-backed stores through `MemoryStoreService`. The distinction is:

* memory stores are in-memory and not committed to disk
* transient stores are also in-memory, but are specifically reset each block as part of the execution lifecycle

## KVStore Wrappers

### CacheKVStore

`cachekv.Store` wraps a `KVStore` with cached reads and buffered writes:

```go expandable
package cachekv

type GStore[V any] struct {
	cache  map[string]*cValue[V]
	parent types.GKVStore[V]
}

func (store *GStore[V]) Write() {
	// flush dirty cached values into the parent store
}

func (store *GStore[V]) CacheWrap() types.CacheWrap {
	return NewGStore(store, store.isZero, store.valueLen)
}
```

This is the wrapper that gives you a working set on top of another store. Reads can be satisfied from cache, writes are buffered, and `Write()` flushes the dirty entries to the parent store.

You usually rely on `cachekv` indirectly through `CacheWrap()` or `CacheMultiStore()`. It is the wrapper that makes nested branching practical.

### `GasKv` Store

`gaskv.Store` charges gas for reads, writes, deletes, and iteration:

```go expandable
package gaskv

type GStore[V any] struct {
	gasMeter  types.GasMeter
	gasConfig types.GasConfig
	parent    types.GKVStore[V]
}

func (gs *GStore[V]) Get(key []byte) (value V) {
	gs.gasMeter.ConsumeGas(gs.gasConfig.ReadCostFlat, types.GasReadCostFlatDesc)
	value = gs.parent.Get(key)
	return value
}
```

Most developers use `gaskv` indirectly. `sdk.Context.KVStore`, `sdk.Context.TransientStore`, and `sdk.Context.ObjectStore` wrap the underlying stores with gas metering before returning them. This means normal keeper operations are already charged correctly as long as they access state through the context or service-backed abstractions.

### `Prefix` Store

`prefix.Store` exposes a logical subspace of another store by automatically prefixing keys:

```go expandable
package prefix

type GStore[V any] struct {
	parent types.GKVStore[V]
	prefix []byte
}

func (s GStore[V]) Get(key []byte) V {
	return s.parent.Get(s.key(key))
}
```

This is commonly used to partition a module store into smaller namespaces without allocating a separate mounted substore.

### `ListenKv` Store

`listenkv.Store` is the wrapper used for state listening and streaming. It records writes to a `MemoryListener`:

```go expandable
package listenkv

type Store struct {
	parent         types.KVStore
	listener       *types.MemoryListener
	parentStoreKey types.StoreKey
}

func (s *Store) Set(key, value []byte) {
	s.parent.Set(key, value)
	s.listener.OnWrite(s.parentStoreKey, key, value, false)
}

func (s *Store) Delete(key []byte) {
	s.parent.Delete(key)
	s.listener.OnWrite(s.parentStoreKey, key, nil, true)
}
```

This wrapper is mostly infrastructure-facing. If you are integrating state streaming or other write listeners, `listenkv` is the mechanism that records writes as they pass through the store stack.

Most module authors do not use `listenkv` directly, but it is useful to know that listeners attach at the store wrapper layer, not inside module keeper code.

## `BasicKVStore` interface

`BasicKVStore` is the minimal subset of the store API:

```go expandable
package types

type BasicKVStore = GBasicKVStore[[]byte]
```

It only includes:

* `Get`
* `Has`
* `Set`
* `Delete`

When a caller only needs simple point reads and writes, depending on `BasicKVStore` instead of full `KVStore` can make that intent explicit.
